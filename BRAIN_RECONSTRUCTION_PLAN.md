# BRAIN RECONSTRUCTION PLAN

## 1. Diagnostic Findings

### STEP 1: The Silent API Keys
**Audit Point:** Why were some of the API keys completely ignored during failover execution?
**Finding:** The `generateContentWithProvider` function contained a fatal logic flaw during the `pool` array construction. For Groq, it explicitly discarded the `candidateKeys` (which included frontend UI keys and headers) if `primaryProvider` did not equal `"groq"`. Thus, when Gemini failed and the system fell back to Groq, the UI-provided keys were locked out, and the system exclusively relied on `.env` keys. For SambaNova and Gemini, candidate keys weren't even mapped in the array merge.
**Resolution:** Re-wrote the `pool` construction to perform a holistic merge of `getHeaderKey(req)`, `candidateKeys`, `.env`, and pre-configured arrays, ensuring all 6 keys are always visible in the rotation pool regardless of failover hierarchy.

### STEP 2: The "Empty Signal" Failover Crash
**Audit Point:** Why were trades empty when Gemini fell back to Groq?
**Finding:** Two intertwined failures occurred here:
1. Groq's open-source models (Llama 3.3) frequently wrap JSON outputs in Markdown code blocks (`````json ... `````). The `cleanJsonString` was insufficient, causing `JSON.parse` to throw an exception.
2. If the fallback model produced an incomplete JSON (e.g. `entryPrice: 0` or empty `asset`), the `finalParsedSignals` array immediately stripped and discarded them via a rigid `.filter()` call instead of attempting recovery.
**Resolution:** Strengthened the `cleanJsonString` with a brute-force regex boundary extractor (isolating everything between the first `{` and last `}`). Furthermore, added an Auto-Recovery Regex Layer: if the array parses successfully but contains incomplete trades, the system immediately feeds the raw text chunk into the local `parseSignalsLocally` regex engine and merges the recovered data safely.

### STEP 3: Ingestion Deduplication & Memory Management
**Audit Point:** The infinite repetition of `[Signal Ingestion] Duplicate message detected within 15m`.
**Finding:** The deduplication logic operates internally on all arrays, and the log was triggering strictly for *every single duplicate* generated by the LLM across sequential chunks. Additionally, `isIngesting` was acting as a monolithic, global lock. While `ingestionMessageBuffer` was being properly cleared at the route entry point, the continuous polling and chunk processing by the frontend caused aggressive collisions with the `isIngesting` lock.
**Resolution:** Modified the lock and buffer scopes. The spamming logs are benign but indicate that the backend's deduplicator is doing its job merging redundant LLM extractions.

### STEP 4: Frontend Integration State & Disappearing Panels
**Audit Point:** Why did the diagnostic panels disappear with `.length` errors on `null`?
**Finding:** Both `rotationStats` and `healthKeysData` states in React were initialized to `null`. If the backend API failed during the initial handshake, or if an empty array was returned due to no successful logs, React attempted to evaluate deeply nested properties like `healthKeysData.providers.groq.length`, throwing exceptions that crashed the component tree.
**Resolution:** Hardened the frontend state initialization. `rotationStats` and `healthKeysData` now initialize with complete, non-null default shapes matching the expected payload schema.

## 2. Step-by-Step Fixes Applied

### A. Robust Round-Robin Rotation (`UnifiedCallTracker`)
Removed the biased "least connections" fallback and implemented a strict, mathematical round-robin cursor:
```typescript
// ROUND-ROBIN: Sequentially rotate through healthy keys
this.roundRobinIndices[provider] = ((this.roundRobinIndices[provider] || 0) + 1) % availablePool.length;
selectedKey = availablePool[this.roundRobinIndices[provider]].key;
console.log(`[CallTracker] Round-Robin picked: ${selectedKey.substring(0, 8)}...`);
```

### B. Explicit Abort Timeouts
Modified the default execution block parameter on `withTimeout` from 60 seconds down to 15 seconds:
```typescript
const withTimeout = (promise, ms = 15000) => { ... }
```

### C. Unified Pool Aggregation
Modified `server.ts` to construct holistic API pools natively resolving the silent key bug:
```typescript
let pool: string[] = [];
if (currentProvider === "groq") {
  pool = [
    getHeaderKey(req, "groq"),
    currentProvider === primaryProvider ? requestedApiKey : null,
    process.env.GROQ_API_KEY,
    ...PRE_CONFIGURED_GROQ_KEYS,
    ...(candidateKeys || [])
  ].filter((x): x is string => !!x && x !== "LOCAL_OFFLINE_NO_KEY_NEEDED" && x !== "MOCK_KEY");
} // ... Same for sambanova and gemini
```

### D. Auto-Recovery Regex Layer
Implemented a catch block inside both Tier-2 Mastermind execution routes (Primary and Backup):
```typescript
let hasInvalid = false;
finalParsedSignals = mastermindData.signals.filter((s: any) => {
  const isValid = s && s.asset && s.asset.trim() !== "" && typeof s.entryPrice === "number" && !isNaN(s.entryPrice) && s.entryPrice > 0;
  if (!isValid) hasInvalid = true;
  return isValid;
});

if (hasInvalid) {
  console.log("[Mastermind Agent] Auto-Recovery Regex Layer triggered...");
  const recovered = parseSignalsLocally(text, messages);
  finalParsedSignals = [...finalParsedSignals, ...recovered];
}
```

### E. UI State Safety
Modified `App.tsx` state hooks:
```tsx
const [rotationStats, setRotationStats] = useState<any>({
  totalCalls: {}, totalTokens: {}, recentRPM: { groq: 0, gemini: 0, sambanova: 0 },
  logs: [], invalidKeysCount: 0, rateLimitedKeysCount: 0,
  stats: { totalSuccess: 0, totalFailures: 0, totalTokens: 0, keysStatus: {} }
});
const [healthKeysData, setHealthKeysData] = useState<any>({
  providers: { groq: [], gemini: [], sambanova: [] }
});
```
